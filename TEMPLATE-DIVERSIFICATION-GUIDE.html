PetTransportGlobal — Template Diversification & Indexation Build Guide

This document is fed into the build chat. It instructs the agent to stop using a single Hugo template across thousands of nearly identical route pages, introduce 5 distinct templates with 5 content-structure souls, rotate them deterministically across the site, and back-fill existing pages so Google indexes them as distinct documents rather than templated boilerplate.

The problem in one line. ~5,000 route pages, 75 country pages, plus airline / breed / origin pages all share one Hugo layout per content type. Same DOM order, same H2 sequence, same sidebar, same FAQ block position, same data-table location. That is a template footprint Google can fingerprint and use to throttle crawl + skip indexation.

Repetition fingerprint (route pages)

Every routes/single.html page emits, in this exact order:

  1. Jumbotron hero with AR → AU badge pair
  2. "Get Your Quote" yellow accent card
  3. Overview paragraph
  4. <h2>Step-by-Step Process</h2> — numbered cards
  5. <h2>{country} Import Requirements</h2> — 6-cell grid
  6. <h2>Exporting from {country}</h2>
  7. <h2>Cost Factors</h2>
  8. <h2>Breed Restrictions</h2> (where present)
  9. <h2>Important Warnings</h2>
  10. FAQ accordion
  11. Right sidebar with on-page nav + route summary + related routes + quote CTA

5,000 pages, identical skeleton, identical microformats, identical CSS class sequence. Google's near-duplicate detector sees a content cluster, samples it, and indexes a tiny fraction.

2. Why one template causes deindexation

Three distinct signals pile up:

  • Boilerplate ratio. Per route, the unique text is roughly 600–900 words. The shared template chrome plus repeated section labels is more than half the rendered HTML. Pages tip below the unique-content threshold.
  • DOM template fingerprint. Same nesting, same class names, same H2 sequence. Trivial to cluster.
  • Crawl budget waste. If Google decides 100 sampled routes look the same, it stops fetching the rest. The famous GSC status "Discovered, not indexed" follows.

3. Design constraint: same brand, different skeletons

All 5 templates must:

  • Use the existing palette (primary blue, secondary gold, light/dark cards already in maincolors.css).
  • Use the existing fonts (Roboto + Montserrat).
  • Pass the existing accessibility + mobile-first checks.
  • Continue to render the same data fields from the route JSON. Nothing is dropped.

What changes:

  • Section order.
  • Section visual treatment (cards vs tables vs side-by-side vs accordion vs stepper).
  • Hero composition (full-width image vs split vs minimalist banner vs map vs data-strip).
  • Sidebar presence and content (left, right, top, none, sticky-bottom).
  • FAQ placement and format (accordion mid-page, two-column block, inline Q&A, expander grid, end-of-page list).
  • Internal-link block position and visual style.

4. The 5 route templates

Each template gets its own Hugo layout file in site/layouts/routes/ and its own scoped CSS block in site/assets/css/main.css under a single root class.

Template A — Field Manual

Layout file: route-a-fieldmanual.html

Hero: Slim banner, two-line title, route badges left-aligned, complexity + weeks chip strip beneath.

Sidebar: None. Full-width single column.

Section order: Hero → Overview → Documents checklist (top, not buried) → Timeline (vertical stepper) → Import requirements (table, not card grid) → Export requirements (table) → Costs (single line + breakdown table) → Warnings (red-bordered call-outs) → FAQ accordion → Related routes inline grid (bottom).

Visual signature: Tabular, dense, no big icons. Looks like an operations manual.

Template B — Visual Journey

Layout file: route-b-journey.html

Hero: Full-bleed image, centered title, big animated route arrow with origin/destination flag emojis. CTA button row.

Sidebar: Right sidebar with sticky route summary card and "what other owners did" block.

Section order: Hero → Quick-answer summary box (3 bullet facts) → Timeline as horizontal scroll cards → Import requirements as 2-col card grid → Cost calculator widget (range slider, JS) → Export requirements (collapsible) → Breed restrictions (alert) → Warnings (chip strip, not boxes) → FAQ → Inline quote form.

Visual signature: Marketing-style, image-led, larger negative space.

Template C — Comparison Brief

Layout file: route-c-comparison.html

Hero: Split hero. Left half is origin (flag, name, exit summary). Right half is destination (flag, name, entry summary). Title bar bridges them.

Sidebar: Left rail with vertical step indicator (1–8 dots tracking scroll).

Section order: Hero → Side-by-side compare (export vs import in one table) → Single timeline list → Cost table → Airlines that fly the route (logo grid, real names) → Documents checklist (split: origin docs vs destination docs) → Warnings → FAQ as Q-then-A blocks (no accordion) → Related routes.

Visual signature: Side-by-side comparisons. Heavy use of two-column layouts.

Template D — Conversational Q&A

Layout file: route-d-qa.html

Hero: Minimal. White background, large H1 phrased as a question ("Moving your dog from {origin} to {destination}?"), short answer paragraph beneath, single CTA.

Sidebar: None on desktop, floating CTA bar at bottom on mobile.

Section order: Hero → "What does this involve?" Q+A → "How long does it take?" Q+A with timeline embedded inside the answer → "What does it cost?" Q+A with cost table inside the answer → "What documents do I need?" Q+A with checklist inside → "Which airlines?" Q+A → "What about breed restrictions?" Q+A → "What can go wrong?" Q+A → "What do I do next?" Q+A with quote CTA.

Visual signature: Each section is structurally a question, not a noun. Eliminates "Step-by-Step Process" / "Import Requirements" templated H2 fingerprint entirely.

Template E — Data-Forward

Layout file: route-e-data.html

Hero: Stats strip hero. Origin name, destination name, 4 large stats (typical weeks, complexity, lowest cost, restricted breeds count). No image.

Sidebar: Right sidebar with TL;DR and downloadable checklist link.

Section order: Hero stats → TL;DR paragraph → All requirements as ONE master table (column = requirement, column = origin rule, column = destination rule, column = source) → Timeline as Gantt-style horizontal bar → Cost as itemised invoice mock → Airlines as data table with weight limits column → Warnings → Linked guides → FAQ at the very bottom.

Visual signature: Looks like a research brief. Tables dominate. Reads like a data sheet.

6. Content structure souls (5 per silo)

Templates change the layout. Souls change the writing structure. Both must rotate together. Souls are markdown files the chat reads before writing the page body. Path: workforce/content/souls/.

Route souls (paired with templates A–E)

Soul 1 — The Operator

Default partner: Template A

Voice: senior pet logistics coordinator. Direct, sequential, procedural. No marketing softeners. Opens with a single declarative sentence stating route status (open / restricted / quarantine-mandatory). Uses second-person sparingly. Terminology: "consignment", "manifested cargo", "endorsement", "AVA", "DAFF", "APHIS" — actual industry vocabulary.

Section openers (must vary across pages): "First step.", "Before any of this matters,", "The order matters here:", "What most owners get wrong:", "Realistic timing:".

Banned: "ultimate guide", "everything you need to know", "in this article we will".

Soul 2 — The Owner-Story

Default partner: Template B

Voice: an owner who has done this route, writing for the next owner. Mixes lived detail ("the vet in Buenos Aires made us re-do the titre because the lab wasn't on the AQIS approved list") with concrete process. First-person plural is allowed sparingly.

Opens with a moment ("Three months out, we thought we had everything sorted.") not a definition.

Banned: bullet lists in the first 200 words. Story first, then structure.

Soul 3 — The Comparison Analyst

Default partner: Template C

Voice: contrasts origin and destination at every turn. Sentences habitually use "whereas", "by contrast", "the difference at this stage is". Opens with the single biggest delta between the two countries (often quarantine or rabies titre).

Eliminates lone facts. Every fact about origin is paired with the corresponding fact about destination.

Soul 4 — The Conversational Advisor

Default partner: Template D

Voice: writes in answers, not declarations. Each section literally begins with the question a worried owner would ask. Uses "you" and "your" naturally. Short paragraphs (2–3 sentences max). No H2s phrased as nouns.

Mandatory pattern: every numeric claim is followed by a one-sentence "in plain terms" gloss.

Soul 5 — The Brief Writer

Default partner: Template E

Voice: dry, evidence-led, source-cited. Writes like a country risk briefing. Each major claim has a parenthetical source ("(IATA Live Animals Regs, Ed. 50)" / "(Australia DAFF, current schedule)"). TL;DR up top, three sentences.

Almost no adverbs. No CTAs in body — just the sidebar handles conversion.

Country souls (paired with templates C-A through C-E)

  1. The Country Profile — encyclopaedic, neutral, leads with regulatory status. Pairs with C-A.
  2. The Route Hub Curator — leads with "from here you can move pets to" matrix. Pairs with C-B.
  3. The Quarantine Specialist — quarantine-first framing, even for countries without quarantine ("Why this country is unusually straightforward"). Pairs with C-C.
  4. The Local Storyteller — narrative, lived, regional. Pairs with C-D.
  5. The Practical Checklist — every paragraph reduces to a checkbox. Pairs with C-E.

7. Deterministic rotation algorithm

Rotation must be reproducible (same page always gets the same template, so re-builds don't churn). Use the page slug.

{{/* In Hugo: layouts/routes/single.html becomes a router */}}
{{ $slug := .File.ContentBaseName }}
{{ $hash := 0 }}
{{ range $i, $c := split $slug "" }}
  {{ $hash = add $hash (mul (printf "%d" (int (printf "%X" (.))) | int) (add $i 1)) }}
{{ end }}
{{ $variant := mod $hash 5 }}
{{ if eq $variant 0 }}{{ partial "routes/route-a-fieldmanual.html" . }}
{{ else if eq $variant 1 }}{{ partial "routes/route-b-journey.html" . }}
{{ else if eq $variant 2 }}{{ partial "routes/route-c-comparison.html" . }}
{{ else if eq $variant 3 }}{{ partial "routes/route-d-qa.html" . }}
{{ else }}{{ partial "routes/route-e-data.html" . }}
{{ end }}

Better: assign template_variant in the route's front matter when the page is generated, so chat can override per-page if needed.

---
template_variant: "C"
content_soul: 3
---

And the layout becomes:

{{ $variant := .Params.template_variant | default "A" }}
{{ partial (printf "routes/route-%s.html" (lower $variant)) . }}

8. Per-page workflow the chat must follow

This is the loop the build agent runs for every page touched. Do not deviate.
  1. Read the front matter. Note template_variant and content_soul. If absent, compute from slug hash and write back.
  2. Load the soul file from workforce/content/souls/route-soul-{N}.md. Read it in full before writing any prose.
  3. Open one nearby existing page with the same template_variant. Compare opening sentences. The new page's opener must not match the prior one's structure (use the soul's "section openers" list).
  4. Write or rewrite the body following the soul's structure rules. The Wordsmith handles voice. The Chameleon (humaniser) runs after.
  5. Verify the H2 set. The set of H2 strings on the page must come from the template's section list, not the universal default list. No H2 may appear identically on more than 30% of pages in that template's bucket.
  6. Verify the FAQ block. At least 2 of the 6 FAQs must be route-specific (mention origin or destination by name and a route-specific number).
  7. Internal links. Use the rotation in §7 also for the related-route block: pull links from a different template bucket so the related-page previews look visually different.
  8. Build + spot-render with hugo --gc --minify. Verify no template errors.
  9. Mark page done in the back-fill register (see §10).

CSS isolation

Every variant CSS file is scoped under a single root class (e.g. .tpl-route-a {…}). The partial wraps everything in <article class="tpl-route-a">. No bleed.

Schema

Schema.org JSON-LD must vary too. Templates A and E emit HowTo. Template C emits HowTo + ComparisonTable microdata. Template D emits FAQPage as the dominant entity. Template B emits Service + HowTo. Diversity here is a meaningful signal.

10. Phased back-fill plan

5,000 routes cannot be re-edited in one session. Tier them.

TierWhatPagesAction
T0Top-traffic routes (GSC top 200 by impressions)~200Manual rewrite by chat against assigned template + soul. One per chat turn.
T1Indexed but low-impression~600Programmatic: assign variant in front matter, regenerate body via chat in batches of 25.
T2"Discovered, not indexed" pages~unknown until GSC pullHighest priority for Template D or E (most differentiated). Rewrite body 60%+.
T3Long-tail routesremainderAdd front-matter variant; let layout handle the visual change; touch body lightly.

Back-fill register

Maintain data/backfill_register.json:

{
  "argentina-to-australia": {
    "template_variant": "C",
    "content_soul": 3,
    "rewritten_at": "2026-05-08",
    "h2_unique_pct": 0.83,
    "boilerplate_ratio": 0.41
  }
}

11. QA, fingerprint test, indexation monitoring

Fingerprint test (run weekly)

Script: scripts/fingerprint_audit.py (to be created).

  1. Crawl each rendered page in public/.
  2. Extract the H2 sequence, the class-name sequence (DOM skeleton hash), the visible-text length, and the boilerplate-text length.
  3. Cluster pages by skeleton hash. Pass: no cluster contains more than 25% of pages in its silo. Fail: any cluster > 40%.
  4. Report H2 strings appearing on >30% of pages within a template variant. These must be paraphrased.

Indexation monitoring

  • Pull GSC Index Coverage weekly. Track "Discovered, not indexed" + "Crawled, not indexed" counts.
  • Sample 50 pages per template variant via the URL Inspection API monthly.
  • When a variant outperforms the others on indexation rate, increase its share in the rotation by 5% in the next quarterly pass.

12. Hard rules (do not violate)

Absolute don'ts

  • Do not collapse two templates into one because "they look similar". They must remain structurally distinct.
  • Do not introduce a 6th template. Keep the rotation at 5. Drift erodes the design system.
  • Do not let any template's share fall below 15% or rise above 25%.
  • Do not write H2s like "Step-by-Step Process" identically across >30% of pages within a template bucket. Paraphrase.
  • Do not reuse the FAQ block from partials/faq-accordion.html on every template. Use the variant matched to the template.
  • Do not skip the soul file load step. The chat must read the soul before writing prose.
  • Do not back-fill more than 25 pages in a single chat session. Quality dies.

Absolute musts

  • Front matter on every route must include template_variant and content_soul.
  • Every page rewrite logs to data/backfill_register.json.
  • Every variant CSS is scoped under .tpl-route-{x}.
  • The fingerprint audit must run before every Surge deploy.
  • Schema.org type varies per template (see §9).
